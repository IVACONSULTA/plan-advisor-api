# Railway Deployment Guide — PA Plan Advisor API

> **App 2 — Backend Express API**  
> Deploys to Railway as a persistent HTTP service connected to PostgreSQL, a Railway Volume, and two internal Python agents.

**Dev deployment (public API base URL):** `https://planadvisor-dev.up.railway.app` — use this host in curl examples, Postman, and any client that calls the API. (`FRONTEND_URL` in Railway should still be your **web app** origin for CORS, e.g. Netlify — not this API URL.)

## 1. Prerequisites

Before starting, make sure you have:

- A [Railway](https://railway.com/) account (free tier works for initial setup)
- The `railway` CLI installed:
  ```bash
  npm install -g @railway/cli
  railway login
  ```
- This repository cloned locally and pushed to GitHub
- A [Supabase](https://supabase.com/) project with **Email/Password auth enabled** (used only for JWT validation — no Supabase DB)
- Your Supabase **Project URL** and **anon key** ready (Settings → API)

## 3. Add the PostgreSQL Service

The API connects to Railway's managed PostgreSQL using the `DATABASE_URL` variable that Railway injects automatically.

### Steps

1. In the Railway project canvas, click **+ New Service**
2. Select **Database → PostgreSQL**
3. Railway provisions a PostgreSQL 15 instance and names it `Postgres`
4. Click the `Postgres` service → **Variables** tab
5. Copy the `DATABASE_URL` value — you'll need it in step 4 to run the schema

### What Railway injects automatically

When your Express API service is added to the same project, Railway automatically injects these variables into the API service environment:

| Variable | Example value |
|---|---|
| `DATABASE_URL` | `postgresql://postgres:xxx@postgres.railway.internal:5432/railway` |
| `PGHOST` | `postgres.railway.internal` |
| `PGPORT` | `5432` |
| `PGDATABASE` | `railway` |
| `PGUSER` | `postgres` |
| `PGPASSWORD` | `<auto-generated>` |

> **Important:** The `.railway.internal` hostname is only accessible from within the same Railway project over the private network. This means zero egress cost and sub-millisecond latency between the API and the database.

## 4. Run the Database Schema

The schema file at `db/schema.sql` creates all 13 tables. Run it once against the Railway PostgreSQL instance.

### Option A — Using the Railway CLI (recommended)

```bash
# Link your local project to Railway
railway link

# Run the schema directly against the Railway database
railway run psql $DATABASE_URL -f db/schema.sql
```

### Option B — Using the Railway Dashboard Query Editor

1. Click the `Postgres` service in the dashboard
2. Go to the **Query** tab
3. Paste the contents of `db/schema.sql`
4. Click **Run**

### Option C — Using `psql` locally with the public DATABASE_URL

1. In the `Postgres` service → **Connect** tab, copy the **Public URL** (for external access)
2. Run:
   ```bash
   psql "postgresql://postgres:xxx@monorail.proxy.rlwy.net:PORT/railway" -f db/schema.sql
   ```

### Verify the tables were created

```sql
-- Run in the Railway Query tab or via psql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;
```

Expected output — 13 tables:

```
 assumptions
 audit_logs
 ai_usage_logs
 calculation_profiles
 companies
 countries
 document_analyses
 documents
 plans
 providers
 scenarios
 transaction_rules
 users_profile
```

## 5. Add the Railway Volume

Documents uploaded by the Admin are stored in a persistent Railway Volume mounted at `/data/documents` in the Express API container.

### Steps

1. In the Railway canvas, click the **Express API service** (after deploying it in step 6)
2. Go to **Settings → Volumes**
3. Click **+ Add Volume**
4. Set:
   - **Mount path:** `/data/documents`
   - **Size:** start with `5 GB` (adjust as needed — costs $0.25/GB/month)
5. Save — Railway will redeploy the service with the volume mounted

> The `DOCUMENTS_PATH` env variable in the API defaults to `/data/documents`. If you use a different mount path, update `DOCUMENTS_PATH` in the service variables.

### Volume structure

The API automatically creates subdirectories per profile:

```
/data/documents/
├── <profile_id_1>/
│   ├── pricing_b2brouter_fr.pdf
│   └── guide_einvoicing_fr.docx
└── <profile_id_2>/
    └── tariff_sheet_es.xlsx
```

## 6. Deploy the Express API

### Option A — Deploy from GitHub (recommended)

1. In the Railway canvas, click **+ New Service → GitHub Repo**
2. Connect your GitHub account if not already connected
3. Select the `PlanAdvisorAPI` repository
4. Railway reads `railway.json` and builds with **Railpack** (Node 22+ when using the repo Dockerfile base image, or per `engines` / `nixpacks.toml`; `npm install`, then `npm start`)
5. The service runs **`npm start`** → `node server.js` (`dotenv` is loaded from code at startup)

### Option B — Deploy via CLI

```bash
# From the project root
railway up
```

### railway.json (already in the repo)

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "RAILPACK"
  },
  "deploy": {
    "healthcheckPath": "/live",
    "healthcheckTimeout": 60,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 3
  }
}
```

Railway’s **deploy health check** calls **`GET /live`** — instant **HTTP 200** with `{ "status": "ok" }` so the probe never waits on Postgres.

For ops and dashboards, **`GET /health`** still returns **HTTP 200** once the API is listening; use the JSON body for database readiness:

```json
{
  "status": "ok",
  "db": "connected",
  "agents": { "doc_agent": true, "summary_agent": true },
  "timestamp": "2026-04-29T10:00:00.000Z"
}
```

If Postgres is unreachable, you may see `"status": "degraded"` and `"db": "unreachable"` (**still HTTP 200** — the probe used at deploy time is **`/live`**, not **`/health`**).

The root **`Dockerfile`** remains in the repo if you prefer a container build; switch `railway.json` `"build"."builder"` to **`DOCKERFILE`** and set **`dockerfilePath`** to **`Dockerfile`** when you want that path instead of Railpack.

## 7. Set Environment Variables

In the Railway dashboard → Express API service → **Variables** tab, add:

### Required variables

| Variable | Value | Notes |
|---|---|---|
| `DATABASE_URL` | *(auto-injected by Railway)* | Do not set manually — Railway links it |
| `SUPABASE_URL` | `https://xxxx.supabase.co` | From Supabase → Settings → API |
| `SUPABASE_ANON_KEY` | `eyJ...` | From Supabase → Settings → API |
| `DOCUMENTS_PATH` | `/data/documents` | Must match your Volume mount path |
| `AGENT_API_KEY` | `<strong-random-secret>` | Shared secret for internal service calls |
| `PA_PLAN_API_KEY` | `<strong-random-secret>` | Clients send **`X-API-Key`** with this value on **all** routes except `GET /live` and `GET /health`. **Required** when `NODE_ENV=production`. |
| `NODE_ENV` | `production` | Enables SSL for PostgreSQL |
| `PORT` | `3000` | Railway sets this automatically too |
| `FRONTEND_URL` | `https://your-app.netlify.app` | For CORS — your Netlify URL |
| `TRUST_PROXY_COUNT` | *(omit = default)* | Express **`trust proxy`** hop count; **1** is baked in when unset (correct for Railway’s edge). Set only if you add another reverse proxy in front. |

### Internal service URLs (set after deploying agents)

| Variable | Value | Notes |
|---|---|---|
| `DOC_AGENT_URL` | `http://pa-doc-agent.railway.internal:8000` | Set after deploying App 3 |
| `SUMMARY_AGENT_URL` | `http://pa-summary-agent.railway.internal:8001` | Set after deploying App 4 |

> **Tip:** Railway lets you reference other service variables with `${{Postgres.DATABASE_URL}}` syntax in the Variables tab, so the database URL stays in sync automatically if the service is renamed.

### Set variables via CLI

```bash
railway variables set NODE_ENV=production
railway variables set SUPABASE_URL=https://xxxx.supabase.co
railway variables set SUPABASE_ANON_KEY=eyJ...
railway variables set DOCUMENTS_PATH=/data/documents
railway variables set AGENT_API_KEY=$(openssl rand -hex 32)
railway variables set PA_PLAN_API_KEY=$(openssl rand -hex 32)
railway variables set FRONTEND_URL=https://your-app.netlify.app
```

## 8. Connect to Supabase Auth

Supabase is used **only for authentication** — to validate JWTs and verify user identity. No business data lives in Supabase.

### Supabase setup

1. Go to [supabase.com](https://supabase.com/) → your project
2. **Authentication → Providers** → Enable **Email** provider
3. **Settings → API** → Copy:
   - **Project URL** → `SUPABASE_URL`
   - **anon public key** → `SUPABASE_ANON_KEY`

### How it works in the API

```
Frontend login (Supabase signInWithPassword)
  → Supabase issues a JWT
  → Frontend sends JWT in Authorization: Bearer <token>
  → Express API validates JWT with supabase.auth.getUser(token)
  → Role is read from Railway PostgreSQL users_profile.role (never from the JWT)
```

### Create the first Admin user

After both Supabase and Railway are live:

**1. Create the user in Supabase Auth (Dashboard → Authentication → Users → Invite User)**

Copy the generated UUID from the user row.

**2. Insert the users_profile in Railway PostgreSQL:**

```bash
# Via railway CLI
railway run psql $DATABASE_URL -c "
  -- First create the internal company
  INSERT INTO companies (name, type) VALUES ('Your Company', 'internal');

  -- Then create the admin profile
  INSERT INTO users_profile (id, email, full_name, role, company_id)
  VALUES (
    '<supabase-uuid-here>',
    'admin@yourcompany.com',
    'Admin User',
    'admin',
    (SELECT id FROM companies WHERE name = 'Your Company')
  );
"
```

**Or use the API endpoint (once deployed):**

```bash
curl -X POST https://planadvisor-dev.up.railway.app/api/admin/users \
  -H "Authorization: Bearer <admin-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "supabase_id": "<uuid>",
    "email": "user@example.com",
    "full_name": "New User",
    "role": "client",
    "company_id": "<company-uuid>"
  }'
```

## 9. Verify the Deployment

### Health check

```bash
curl https://planadvisor-dev.up.railway.app/live
# Expected: HTTP 200 immediately — `{ "status": "ok" }` (what Railway probes during deploy)

curl https://planadvisor-dev.up.railway.app/health
# Expected: HTTP 200 — body includes "status" ("ok" or "degraded") and "db" ("connected" or "unreachable")
```

### Test authentication

```bash
# 1. Get a JWT from Supabase
TOKEN=$(curl -s -X POST \
  'https://xxxx.supabase.co/auth/v1/token?grant_type=password' \
  -H "apikey: <anon-key>" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"yourpassword"}' \
  | jq -r '.access_token')

# 2. Call a protected endpoint
curl https://planadvisor-dev.up.railway.app/api/countries \
  -H "Authorization: Bearer $TOKEN"
```

### Test the calculator (end-to-end smoke test)

```bash
# 1. Create a country
curl -X POST https://planadvisor-dev.up.railway.app/api/admin/countries \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"code":"FR","name":"France"}'

# 2. Check available countries for calculator
curl https://planadvisor-dev.up.railway.app/api/calculator/available-countries \
  -H "Authorization: Bearer $TOKEN"
```

## 10. Internal Service Networking

The Express API communicates with the two Python agents (App 3 and App 4) over Railway's **private internal network** — no public internet involved.

### How Railway private networking works

- Services in the same Railway project can reach each other via `.railway.internal` hostnames
- No egress cost, sub-millisecond latency
- The service name in the URL is the Railway service name (lowercase, hyphenated)

### Service name → hostname mapping

| Service (Railway name) | Internal hostname | Port |
|---|---|---|
| `pa-doc-agent` | `pa-doc-agent.railway.internal` | `8000` |
| `pa-summary-agent` | `pa-summary-agent.railway.internal` | `8001` |
| `Postgres` | `postgres.railway.internal` | `5432` |

> If you name your Railway service differently, update the `DOC_AGENT_URL` and `SUMMARY_AGENT_URL` environment variables accordingly.

### Architecture diagram

```
Internet
  │ HTTPS (e.g. planadvisor-dev.up.railway.app)
  ▼
Express API (Railway)
  ├──[railway.internal]──► PostgreSQL (Railway)
  ├──[railway.internal]──► Volume /data/documents
  ├──[railway.internal]──► pa-doc-agent (Python/CrewAI)
  └──[railway.internal]──► pa-summary-agent (Python/DSPy)
  │
  └──[public JWT validation only]──► Supabase Auth
```

## 11. Production Checklist

### Before going live

- [ ] `NODE_ENV=production` is set
- [ ] `DATABASE_URL` is linked from the Postgres service (not hardcoded)
- [ ] `AGENT_API_KEY` is a strong random secret (minimum 32 characters)
- [ ] `FRONTEND_URL` is set to your exact Netlify domain (no trailing slash)
- [ ] Volume is mounted at `/data/documents` and size is adequate
- [ ] All 13 tables created — verified with `\dt` in psql
- [ ] At least one `admin` user exists in both Supabase Auth and `users_profile`
- [ ] **`GET /live`** returns `200 OK` (Railway deploy probe)
- [ ] **`GET /health`** returns `200 OK` with `db: connected` once Postgres is healthy

### Security

- [ ] `DATABASE_URL` is only visible in the Railway service — never in frontend code
- [ ] `OPENAI_API_KEY` (used by agents) is set on the agent services, never on this one
- [ ] `storage_path` column is never returned in any API response — confirmed in all queries
- [ ] Railway service is not exposed publicly on ports other than the default HTTP port

### Monitoring

- [ ] Railway **Metrics** tab shows CPU/memory/request graphs for the API service
- [ ] Railway **Logs** tab is bookmarked for debugging
- [ ] Set up a Railway **Alert** on service restart or crash
- [ ] Review `ai_usage_logs` periodically in the Railway Query tab

### Useful Railway CLI commands

```bash
# View live logs
railway logs

# Open a shell in the deployed container
railway shell

# Run a one-off command (e.g. schema migration)
railway run psql $DATABASE_URL -f db/schema.sql

# Redeploy manually
railway up --detach

# Open the Railway dashboard for this project
railway open
```

## Troubleshooting

| Problem | Likely cause | Fix |
|---|---|---|
| `ECONNREFUSED` connecting to PostgreSQL | `DATABASE_URL` not set or wrong | Check Variables tab; Railway auto-injects it only if Postgres is in the same project |
| `401 Unauthorized` on all requests | `SUPABASE_URL` or `SUPABASE_ANON_KEY` wrong | Re-copy from Supabase → Settings → API |
| `403 Forbidden — User profile not found` | JWT is valid but no `users_profile` row exists | Create the row via `POST /api/admin/users` or direct SQL |
| File upload fails | Volume not mounted | Check Settings → Volumes; ensure mount path matches `DOCUMENTS_PATH` |
| Agent call timeout | `DOC_AGENT_URL` / `SUMMARY_AGENT_URL` wrong hostname | Confirm service names in Railway match the `.railway.internal` URLs |
| `relation "users_profile" does not exist` | Schema not run yet | Execute `db/schema.sql` against the Railway PostgreSQL instance |
